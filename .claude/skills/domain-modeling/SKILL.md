# Domain Modeling Skill

새로운 도메인을 설계할 때 사용하는 체계적 모델링 프로세스.
코드를 치기 전에 규칙을 확정하고, 구조를 결정한다.

## 핵심 원칙

1. **코드 전에 규칙**: 요구사항 → 질문 → 규칙 확정 → 경계 결정 → 엔티티 설계 → 코드. 이 순서를 지킨다.
2. **질문으로 규칙을 드러낸다**: 요구사항 문서에 명시되지 않은 암묵적 규칙을 질문으로 끄집어낸다.
3. **결정에는 근거가 있다**: 모든 의사결정에 "왜?"를 붙인다. 근거 없는 결정은 나중에 뒤집힌다.
4. **사용자가 결정한다**: 선택지를 제시하고, 사용자가 선택한다. Claude가 대신 결정하지 않는다.
5. **최소화 지향**: 불필요한 구조를 만들지 않되, 구조가 요구하는 분리는 수용한다.

## Quick Reference

| Phase | 목적 | 핵심 산출물 |
|-------|------|-------------|
| 1. 요구사항 수집 | 도메인의 "무엇"을 파악 | 핵심 개념 목록 + 행위 목록 |
| 2. 질문 기반 규칙 도출 | 암묵적 규칙을 명시화 | Q별 결정 + 판단 근거 |
| 3. 비즈니스 규칙 확정 | 규칙을 카테고리별 정리·확인 | 규칙 테이블 (R1, R2, ...) |
| 4. 도메인 경계 결정 | Aggregate Root·참조 방향 확정 | 경계 결정 3단계 (Q1~Q3) |
| 5. 엔티티 설계 | 코드 진입 직전 설계 확정 | 엔티티 필드 + Enum + 동시성 포인트 |

## Phase 1: 요구사항 수집

사용자가 제공한 입력(PDF, 이미지, 텍스트 등)에서 도메인의 기본 개념을 파악한다.

1. 입력 자료를 읽고 **핵심 개념**을 추출한다 (예: "쿠폰 템플릿"과 "발급된 쿠폰"은 다른 개념)
2. API 스펙이 있다면 **어떤 행위**가 존재하는지 나열한다
3. 체크리스트나 제약 조건이 있다면 **명시된 규칙**을 먼저 정리한다
4. 정리 결과를 사용자에게 보여주고 확인받는다

이 단계에서 코드나 엔티티 구조를 언급하지 않는다. 도메인의 "무엇"에만 집중한다.

---

## Phase 2: 질문 기반 규칙 도출

요구사항 문서에서 **명시되지 않은 암묵적 규칙**을 질문으로 끄집어낸다.

### 질문 도출 방법

요구사항의 각 기능에 대해 아래 관점으로 질문을 만든다:

- **엣지 케이스**: "~하면 어떻게 되는가?" (예: 할인 후 금액이 0원 이하면?)
- **상태 전이**: "A 상태에서 B로 갈 수 있는가? 역방향은?" (예: USED → AVAILABLE 복원이 가능한가?)
- **동시성**: "여러 명이 동시에 ~하면?" (예: 같은 쿠폰을 동시에 사용하면?)
- **삭제/수정 영향**: "X를 삭제하면 X를 참조하는 Y는?" (예: 템플릿 삭제 시 발급된 쿠폰은?)
- **시간 흐름**: "시간이 지나면 상태가 바뀌는가? 언제, 어떻게?" (예: 만료 처리는 즉시? 배치?)
- **소유권/접근**: "누가 이걸 할 수 있는가?" (예: 타인의 쿠폰을 사용할 수 있는가?)

### 질문 제시 형식

한 번에 3~5개씩 질문을 제시한다. 각 질문에는:

- 질문 내용
- 선택지 2~3개 (각각의 장단점 또는 trade-off 설명)
- 추천이 있다면 근거와 함께

사용자가 선택하면 **결정 + 판단 근거**를 기록한다.

### 규칙 간 충돌 체크

모든 질문의 답이 나오면, 규칙 간 충돌이 없는지 교차 검증한다.
충돌이 발견되면 사용자에게 알리고, 어느 쪽을 수정할지 결정받는다.

## Phase 3: 비즈니스 규칙 확정

Phase 2에서 도출한 규칙을 카테고리별로 정리한다.

### 정리 형식

각 규칙은 아래 형태로 기록한다:

```
| # | 규칙 | 근거 |
|---|---|---|
| R1 | [규칙 내용] | [Q몇 결정 / PDF 요구사항 / 개념 정리 등] |
```

카테고리는 도메인에 맞게 자연스럽게 나눈다.
(예: 쿠폰이면 → 템플릿, 발급, 사용, 복원, 만료, 조회)

사용자에게 전체 규칙 목록을 보여주고 최종 확인받는다.

---

## Phase 4: 도메인 경계 결정

확정된 규칙을 기반으로, 3가지 질문을 순서대로 던져서 도메인 경계를 결정한다.
상세 가이드는 `references/boundary-decision-guide.md`를 참조한다.

### 경계 결정 3단계 질문

**Q1. 이 도메인은 어떤 기능을 제공하고, 복잡도는?**
→ 현재 복잡도가 독립 도메인으로의 분리를 정당화하는지 판단한다.

체크포인트:
- 자체 API가 몇 개인가?
- 자체 상태 머신이 있는가?
- 동시성 제어 포인트가 있는가?
- 다른 도메인 없이 독립적으로 의미가 있는가?

**Q2. Aggregate Root는 누구인가?**
→ 도메인 위계(누가 누구를 소유하는가)로 결정한다.

판단 기준:
- null 허용 X: Root 없이 하위 엔티티가 존재할 수 있는가?
- 생명주기 일치: Root가 삭제되면 하위도 함께 삭제되는가?
- 생명주기가 불일치하면 → 별도 Aggregate Root

**Q3. Root가 결정되면, 접근 방향은?**
→ Root를 통해서만 접근하고, 역참조는 원칙적으로 불필요.

판단 기준:
- 비즈니스 책임이 어느 도메인에 있는가?
- 같은 도메인 내부: @ManyToOne 직접 참조 가능
- 도메인 간: Long id만 참조 (결합도 낮춤)
- 역참조가 필요하면: 서비스를 통해 접근

**Q3-검증. 참조 방향이 올바른가?**
→ Q3에서 결정한 모든 도메인 간 참조에 대해, `references/boundary-decision-guide.md`의 **관계 방향 검증 체크리스트**를 적용한다.

4가지 질문(엔티티 자체를 알아야 하는가? / 대부분이 참조하는가? / 이력 관리 책임이 누구에게? / 확장 가능성)을 던져서 "아니오"가 2개 이상이면 방향 역전을 검토한다.

이 검증을 건너뛰면 구현 후 리팩토링이 필요해진다. **반드시 Q3 직후에 수행한다.**

각 질문의 결과를 사용자에게 보여주고 확인받은 뒤 다음 질문으로 넘어간다.

---

## Phase 5: 엔티티 설계

Phase 3(규칙)과 Phase 4(경계)가 모두 확정된 후에 엔티티를 설계한다.

1. 각 Aggregate Root별 엔티티 필드를 나열한다
2. Enum이 필요한 것을 식별한다
3. 동시성 제어 포인트를 명시한다
4. 다른 도메인과의 참조 방향을 명시한다

이 단계에서 비로소 코드 작성으로 넘어갈 수 있다.

---

## 산출물 정리

각 Phase가 끝날 때마다 Notion 등에 기록할 수 있다. 기록 시 아래 구조를 따른다:

```
1. 핵심 개념 구분
2. 확정된 비즈니스 규칙 (카테고리별 테이블)
3. 질문을 통한 규칙 도출 과정 (Q별 결정 + 근거)
4. 도메인 경계 결정 (결정 1, 2, 3)
5. 도메인 경계 의사결정 과정 (Q1, Q2, Q3 + 그 외 고려사항)
6. 다음 단계 플랜
```

의사결정 과정은 교훈이나 감성이 아닌, **실제로 던진 질문 → 비교 → 판단 → 결론**을 사실 그대로 기록한다.

## Common Mistakes

| 실수 | 해결 |
|------|------|
| Phase 2(질문)를 건너뛰고 바로 엔티티 설계 | 반드시 질문으로 암묵적 규칙을 먼저 드러낸다. 엣지 케이스에서 설계가 무너지는 것을 방지 |
| 사용자가 "빨리 코드 짜줘"라고 요청하면 바로 코드 작성 | 규칙이 확정되지 않았으면 규칙부터 잡자고 안내한다. Phase 순서를 지킨다 |
| 한 번에 모든 질문을 한꺼번에 제시 | 3~5개씩 나눠서 제시하여 사용자의 결정 피로를 줄인다 |
| 이전 결정과 충돌하는 새 결정 발견 시 무시 | 충돌을 사용자에게 알리고, 이전 결정 수정을 두려워하지 않는다 |
| Q3(참조 방향) 결정 후 검증 체크리스트 생략 | Q3 직후 반드시 관계 방향 검증 체크리스트를 적용한다. 생략 시 구현 후 리팩토링 필요 |
