# 도메인 경계 결정 가이드

Phase 4에서 사용하는 상세 가이드. 3가지 질문을 순서대로 적용하여 도메인 경계를 결정한다.

---

## Q1. 이 도메인은 어떤 기능을 제공하고, 복잡도는?

### 목적

현재 설계 대상이 독립 도메인으로 분리될 만큼 복잡한지, 아니면 다른 도메인의 하위로 충분한지 판단한다.

### 체크리스트

아래 항목 중 3개 이상 해당하면 독립 도메인을 고려한다:

- [ ] 자체 API가 3개 이상인가?
- [ ] 자체 상태 머신(상태 전이)이 있는가?
- [ ] 동시성 제어 포인트가 있는가?
- [ ] 다른 도메인 없이 독립적으로 CRUD가 가능한가?
- [ ] Admin과 유저 양쪽에서 각각 접근하는가?
- [ ] 배치 처리나 스케줄링 로직이 있는가?

### 판단 기준

- 체크 3개 이상: 독립 도메인으로 분리
- 체크 1~2개: 상위 도메인의 하위로 배치하되, 향후 분리 가능하게 패키지는 분리
- 체크 0개: 상위 도메인에 포함

### 사용자에게 제시할 형식

```
[도메인명]이 제공하는 기능:
- [기능 1]
- [기능 2]
- ...

복잡도 지표:
- 자체 API: N개
- 상태 머신: 있음/없음
- 동시성 포인트: N개
- ...

판단: [독립 도메인 / 하위 도메인]
이유: [근거]
```

## Q2. Aggregate Root는 누구인가?

### 목적

도메인 안에 여러 엔티티가 있을 때, 어떤 엔티티가 Root이고 어떤 엔티티가 하위인지, 또는 각각 독립 Root인지 결정한다.

### 판단 기준: 생명주기 일치 테스트

두 엔티티 A와 B에 대해 아래 3가지를 체크한다:

| 질문 | A가 Root인 경우 | 별도 Root인 경우 |
|---|---|---|
| A 없이 B가 존재할 수 있는가? | 불가 (B.aId NOT NULL) | 불가 (B.aId NOT NULL) |
| B 없이 A가 존재할 수 있는가? | 가능 | 가능 |
| A가 삭제되면 B도 삭제되는가? | **예 → 같은 Aggregate** | **아니오 → 별도 Aggregate** |

세 번째 질문이 핵심이다. A가 삭제(soft든 hard든)되어도 B가 살아있어야 하면, B는 A의 하위가 아니라 독립 Root다.

### 실제 판단 프로세스

1. 도메인 안의 모든 엔티티를 나열한다
2. 각 쌍에 대해 생명주기 일치 테스트를 수행한다
3. 비즈니스 규칙(Phase 3에서 확정한)을 근거로 판단한다
4. 결과를 사용자에게 보여주고 확인받는다

### 사용자에게 제시할 형식

```
[엔티티A]와 [엔티티B]의 생명주기 비교:
- A 삭제 시 B: [살아있어야 함 / 함께 삭제]
  근거: [비즈니스 규칙 R몇]
- A 없이 B 생성: [불가 / 가능]
- B 없이 A 존재: [불가 / 가능]

결론: [같은 Aggregate / 별도 Aggregate Root]
```

### 주의: "NOT NULL이니까 하위다"는 틀린 판단

B.aId가 NOT NULL이라고 해서 B가 반드시 A의 하위 엔티티인 것은 아니다. 생명주기가 다르면 별도 Root다. NOT NULL은 "생성 시 A가 필요하다"는 뜻이지, "A와 함께 죽는다"는 뜻이 아니다.

## Q3. 접근 방향과 참조 방식

### 목적

Aggregate Root가 결정되면, 도메인 내부와 도메인 간의 참조 방향과 방식을 결정한다.

### 참조 방식 결정 기준

| 상황 | 방식 | 이유 |
|---|---|---|
| 같은 도메인, 같은 Aggregate 내부 | @ManyToOne 직접 참조 | 트랜잭션 경계 공유, JPA 편의성 |
| 같은 도메인, 다른 Aggregate | @ManyToOne 직접 참조 가능 | 같은 도메인이므로 결합 허용. 단, 할인 계산 등 실용적 필요가 있을 때 |
| 다른 도메인 간 | Long id만 참조 | 도메인 간 결합도를 낮춤 |

### 역참조 판단

"A → B 방향의 참조가 필요한가?"를 판단할 때:

1. **비즈니스 책임 체크**: B에 대한 모든 비즈니스 로직이 어느 도메인에 있는가?
2. 모든 책임이 B의 도메인에 있으면 → A는 B를 모른다 (역참조 없음)
3. A에서 B 목록이 필요한 유스케이스가 있다면 → B 도메인의 서비스를 통해 접근

### 비즈니스 책임 분석 형식

```
[엔티티]에 대한 비즈니스 책임:

| 행위 | 책임 주체 |
|---|---|
| [행위1] | [도메인A / 도메인B] |
| [행위2] | [도메인A / 도메인B] |
| ... | ... |

→ 모든 책임이 [도메인X]에 있으므로, [엔티티]는 [도메인X]에 속한다.
→ [도메인Y]는 [엔티티]의 존재를 모른다. 필요 시 서비스를 통해 접근.
```

### 다른 도메인에서 결과만 필요한 경우: 스냅샷 패턴

도메인 A가 도메인 B의 "처리 결과"만 필요하고 B의 엔티티 자체는 몰라도 되는 경우, A의 엔티티에 결과를 스냅샷으로 저장한다.

```
Order (주문 도메인)
├── originalAmount — 할인 적용 전 금액
├── discountAmount — 할인 금액 (출처 무관)
├── totalAmount — 최종 결제 금액

OwnedCoupon (쿠폰 도메인)
├── orderId (nullable, Long) — 어떤 주문에서 사용되었는지
```

스냅샷의 장점:
- B 도메인의 데이터가 나중에 변경/삭제되어도 A의 기록은 영향받지 않음
- A가 B 도메인에 대한 의존 없이 자체적으로 완결된 데이터를 가짐
- 조회 시 조인 없이 자체 테이블만으로 충분

### 관계 방향 검증 체크리스트

참조 방향을 결정한 후, 아래 질문으로 방향이 올바른지 검증한다. 이 검증을 건너뛰면 "구현 후 관계 역전" 리팩토링이 필요해진다.

**A → B 참조를 두려고 할 때 던져야 할 4가지 질문:**

| # | 질문 | "아니오"면 참조 방향을 뒤집어야 함 |
|---|---|---|
| 1 | **A가 B의 엔티티 자체를 알아야 하는가?** (결과만 필요한 게 아니라) | 결과만 필요하면 A에 결과를 스냅샷하고, B가 A를 참조해야 함 |
| 2 | **A의 대부분 레코드가 B를 참조하는가?** | 대부분 null이면 관심사가 아닌 것 → A에 둘 필요 없음 |
| 3 | **B의 이력/상태 관리 책임이 A에 있는가?** | B가 자기 이력을 관리해야 하면 B가 A를 참조해야 함 |
| 4 | **B 외에 같은 역할을 하는 C, D가 추가될 수 있는가?** | 확장 시 A에 FK가 계속 늘어나면 방향이 잘못된 것 |

**사용자에게 제시할 형식:**

```
관계 방향 검증: [엔티티A] → [엔티티B]

| 질문 | 답 | 판단 |
|---|---|---|
| A가 B 자체를 알아야 하는가? | [예/아니오 + 근거] | |
| 대부분의 A 레코드가 B를 참조하는가? | [예/아니오 + 근거] | |
| B의 이력 관리 책임이 A에 있는가? | [예/아니오 + 근거] | |
| 같은 역할의 C, D가 추가될 수 있는가? | [예/아니오 + 근거] | |

→ "아니오"가 2개 이상이면 방향 역전을 검토한다.
```

**실제 사례 — 쿠폰-주문 관계 (잘못된 방향 → 수정):**

```
검증 대상: Order → OwnedCoupon (ownedCouponId, nullable)

| 질문 | 답 | 판단 |
|---|---|---|
| Order가 OwnedCoupon 자체를 알아야 하는가? | 아니오 — 할인 결과(discountAmount)만 필요 | ❌ |
| 대부분의 Order가 OwnedCoupon을 참조하는가? | 아니오 — 쿠폰 없는 주문이 대부분 | ❌ |
| OwnedCoupon의 이력 관리 책임이 Order에 있는가? | 아니오 — 쿠폰 사용 이력은 쿠폰 도메인의 책임 | ❌ |
| 쿠폰 외에 포인트, 프로모션 등이 추가될 수 있는가? | 예 — 추가될 때마다 Order에 FK 추가 필요 | ❌ |

→ 4개 모두 "아니오" → 방향 역전 필요
→ 수정: OwnedCoupon.orderId로 역전. Order는 discountAmount만 보유.
```

### 최종 정리 형식

```
참조 방향 규칙:

| 관계 | 방식 | 이유 |
|---|---|---|
| [엔티티A] → [엔티티B] | [@ManyToOne / Long id] | [같은 도메인 / 도메인 간 경계] |
| [엔티티C] → [엔티티A] | 역참조 없음 | [서비스를 통해 접근] |
```

## 그 외 경계 결정 시 던져야 할 질문들

### "테이블을 하나로 할 수 없는가?"

엔티티를 분리하기 전에, 하나로 합칠 수 있는지 먼저 검토한다.
분리가 필요한 신호:
- 다대다(M:N) 관계이면서 중간 테이블에 자기만의 상태가 있다
- 하나의 엔티티가 두 가지 역할(예: 템플릿 + 인스턴스)을 겸하고 있다
- 같은 필드가 레코드마다 다른 의미를 가진다

### "향후 확장이 현재 구조를 막지 않는가?"

당장 구현하지 않을 기능이라도, 현재 구조가 그 확장을 원천적으로 막는지 확인한다.
막지 않으면 → 지금은 구현하지 않는다.
막으면 → 구조를 조정한다 (기능은 구현하지 않되 구조만 열어둔다).

### "기존 규칙과 새 결정이 충돌하지 않는가?"

모든 결정이 끝나면, 전체 규칙과 경계 결정을 교차 검증한다.
충돌이 발견되면 이전 결정을 수정하는 것을 두려워하지 않는다.
전체 정합성이 개별 결정의 최적성보다 중요하다.
