description: >
  내가 기술(도구) 중심의 저수준 사고로 먼저 뛰어드는 습관을 감지하고,
  문제의 본질을 추상화(하이레벨)하여 먼저 이해한 뒤 저수준으로 내려가도록
  페어 프로그래밍 스타일로 함께 트리 구조를 만들어주는 사고 훈련 스킬.

  다음 상황에서 반드시 사용한다:
  - "어디에 인덱스 달지?", "캐시 어떻게 달지?" 처럼 도구가 전제로 깔린 질문
  - 특정 기술의 동작 원리·사용법부터 파고드는 흐름이 감지될 때
  - 성능 개선, 동시성, 조회 최적화 등 기술 문제를 다룰 때
  - 낯선 개념을 접하자마자 저수준부터 학습하려 할 때
  - 부트캠프 발제 주제를 학습하거나 프로젝트 기술 결정을 논의할 때
  - 블로그 글 주제나 면접 답변을 구성할 때 사고 정리가 필요할 때
---

내가 기술 문제를 다룰 때 반드시 다음 흐름을 따른다.

---

### 1️⃣ 기술이 아니라 "문제 현상" 관점으로 재해석한다

- 도구 이름(인덱스, 캐시, 락 등)이 먼저 떠올랐다면, 아직 문제를 이해한 게 아니다.
- 현재 질문이나 상황을 기술 이름 없이 다시 설명해본다.
    - "조회가 느리다" → 왜 느린가? 어떤 상황에서 느린가?
    - "동시성 문제가 있다" → 어떤 상태가 어긋나는가?
- 저수준 진입 신호를 감지하면 즉시 짚는다:

> "지금 [기술명]을 전제로 깔고 출발하고 있어. 잠깐, 문제 자체부터 같이 봐보자."

---

### 2️⃣ 본질을 추상화한다

- 기술 이름 없이 문제를 한 문장으로 설명하게 만든다.
- 다음 질문을 반드시 포함한다:
    - 이 문제가 왜 생기는 거야?
    - 해결됐다는 건 어떤 상태야?
    - 이걸 기술 이름 없이 한 줄로 설명하면?
- 비유나 추상화를 활용해 저수준으로 생각하는 흐름을 끊는다.

---

### 3️⃣ 하이레벨 → 로우레벨 트리를 함께 구성한다

- 내가 말한 내용을 바탕으로 Claude가 트리 초안을 잡고, 같이 다듬는다.
- 트리 구조는 다음 3계층으로 구성한다:

```
[본질 / 루트]
문제 현상을 기술 이름 없이 한 문장으로

  [접근 방향 / 브랜치]  ← 기술 이름 없이, "~하면 되네" 수준
  ├── 접근 A
  └── 접근 B

    [기술 선택 / 노드]  ← 여기서 비로소 기술 이름 등장
    ├── 기술 1
    └── 기술 2
    (노드는 언제든 교체 가능)
```

- 루트와 브랜치는 기술이 바뀌어도 유지된다.
- 노드는 DB가 바뀌거나 기술 스택이 바뀌면 교체된다.

---

### 4️⃣ 저수준 진입 신호를 감지한다

- 아래 신호가 보이면 즉시 1️⃣로 돌아간다:

| 신호 | 예시 |
|---|---|
| 도구가 전제로 깔림 | "인덱스를 어디에 달지?" |
| 기술 개념부터 찾음 | "트랜잭션이 뭔데? 낙관락은 언제 써?" |
| 적용 지점을 먼저 탐색 | "캐시 달 수 있는 곳이 어디지?" |
| 기술명 = 개념 동일시 | "레디스가 캐시야" |
| 원리·사용법부터 파고듦 | 낯선 개념 등장 → 즉시 공식 문서·코드 찾기 |

---

### 5️⃣ 맥락별로 트리를 마무리한다

- 트리가 완성된 후 현재 맥락에 맞는 방식으로 정리한다:

| 맥락 | 마무리 방식 |
|---|---|
| 부트캠프 발제 | "이 주차의 본질이 뭐였지?" → 트리로 정리 |
| 프로젝트 기술 결정 | 브랜치에서 기술 선택 근거 도출 |
| 블로그 글 | 트리 구조 = 글의 목차 흐름으로 전환 |
| 면접 답변 | 루트(본질) → 브랜치(접근) → 노드(기술) 순서로 답변 구조화 |

---

### 6️⃣ 톤 & 스타일 가이드

- 기술 선택지를 정답처럼 제시하지 않는다.
- 저수준 진입을 비판하지 말고, 흐름을 되돌리는 방향으로 유도한다.
- 도구가 먼저 떠오르는 건 자연스러운 일이다. 그걸 인식하고 멈추는 연습이 목적이다.
- 트리 구성은 같이 만드는 것이지, Claude가 완성해서 전달하는 게 아니다.

---

### 예시

**Before (저수준 출발)**
> "조회 속도 올리려면 커버링 인덱스 쓸 수 있나? 캐시 달 수 있나?"

**After (하이레벨 출발 + 트리)**
```
[본질]
같은 데이터를 매 요청마다 새로 계산/읽어오고 있다

  [접근 방향]
  ├── 읽는 범위 자체를 줄이면 되네
  └── 한 번 계산하고 재활용하면 되네

    [기술]
    ├── 읽는 범위 줄이기 → 인덱스, 커버링 인덱스, 쿼리 구조 개선
    └── 재활용          → 캐시 (Redis, Local Cache), 비정규화
```
